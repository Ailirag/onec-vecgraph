# MCP-сервер «1C Config Vector + Graph»

Векторизация и анализ конфигураций 1С:Предприятие из **XML-выгрузки Конфигуратора** (НЕ EDT):
семантический поиск по коду и метаданным + граф зависимостей (включая граф вызовов BSL),
доступ через MCP. Компонент более крупной мультиарендной системы полного цикла
разработки (БА → СА → архитектор → разработчик → ревьюер → тимлид, human-in-the-loop).

---

## 1. Цели и не-цели

**Цели**
- Парсить выгрузку конфигурации в формате Конфигуратора (метаданные + модули BSL).
- Векторизовать тексты (модули, синонимы, комментарии, представления) и структурированные
  метаданные (реквизиты + их **типы**, включая составные и ссылочные; **значения** перечислений
  и предопределённых).
- Строить граф зависимостей: метаданные (типы → объекты, подсистемы, роли/права, подписки,
  планы обмена) **и граф вызовов BSL**.
- Отдавать всё через MCP-инструменты (Streamable HTTP + stdio), мультиарендно.

**Не-цели (MVP)**
- Поддержка формата EDT.
- Подключение к живой базе 1С.
- Хранение/оркестрация артефактов ролей SDLC (предусмотрена точка расширения, см. §13).
- Генерация/правка BSL-кода (сервер — поставщик знаний, не редактор).

**Наша ниша:** единственный, кто делает *все три* строго из XML Конфигуратора —
векторизация (тексты + метаданные с типами и значениями) **и** граф зависимостей **и** граф вызовов
BSL, в едином хранилище Neo4j, мультиарендно по HTTP.

## 4. Архитектура (слои)

```
            ┌─────────────────────────────────────────────┐
            │  MCP-сервер (FastMCP): Streamable HTTP / stdio │
            │  аутентификация + tenant scoping (заголовки)   │
            └───────────────┬─────────────────────────────┘
                            │ инструменты (§10)
      ┌─────────────────────┼──────────────────────────┐
      │            Сервисный слой (запросы)             │
      │  semantic / hybrid / graph / impact / callgraph │
      └─────────────────────┬──────────────────────────┘
                            │ Cypher + vector + fulltext
                    ┌───────▼────────┐
                    │     Neo4j      │  граф + векторы + FTS
                    └───────▲────────┘
                            │ запись (UNWIND батчи)
      ┌─────────────────────┴──────────────────────────┐
      │            Конвейер индексации (CLI)            │
      │  Parser(XML) → BSL-анализатор → GraphBuilder    │
      │           → Chunker → Embeddings                │
      └─────────────────────────────────────────────────┘
```

Компоненты:
1. **CLI-индексатор** — офлайн-разбор каталога выгрузки → Neo4j (полная и инкрементальная).
2. **Парсер метаданных** (lxml, namespace-aware) — XML → доменные модели.
3. **BSL-анализатор** (tree-sitter-bsl) — AST модулей → процедуры/функции/вызовы.
4. **Построитель графа** — модели → узлы/рёбра (Neo4j driver, батчи).
5. **Эмбеддинги** — провайдер (local/cloud) за интерфейсом + чанкер + кеш.
6. **MCP-сервер** (FastMCP) — инструменты поверх сервисного слоя.

## 5. Мультиарендность

- Каждый узел и `:Chunk` несёт `tenant_id` (компания) и `config_id` (конфигурация/`base`|`ext:<имя>`).
- Все запросы **обязательно** фильтруют по `tenant_id`/`config_id`; фильтр берётся из
  аутентифицированного контекста запроса (заголовок, напр. `X-Tenant-Id` / `X-Config-Id`),
  а не из аргументов инструмента (защита от утечки между арендаторами).
- Векторный поиск: пред-фильтрация по арендатору (через filtered vector search новых версий Neo4j)
  либо over-fetch + пост-фильтр по `tenant_id`; на больших объёмах — отдельный label/индекс на конфиг.
- Уникальность узлов — по составному ключу `(tenant_id, config_id, kind, fqname)`.
- **Путь миграции** на сильную изоляцию: тот же доменный код, но запись в отдельную БД
  Neo4j на арендатора (Enterprise/Aura) — `tenant_id` становится именем БД. Слой хранилища
  абстрагирован, чтобы переключение не затрагивало парсер/построитель графа.

## 6. Схема графа

**Метки узлов:** `Configuration`, `Subsystem`, `MetadataObject` (+ под-метка типа:
`Catalog`, `Document`, `InformationRegister`, `AccumulationRegister`, `AccountingRegister`,
`CalculationRegister`, `Enum`, `ChartOfCharacteristicTypes`, `ChartOfAccounts`,
`ChartOfCalculationTypes`, `BusinessProcess`, `Task`, `CommonModule`, `Report`, `DataProcessor`,
`ExchangePlan`, `WebService`, `HTTPService`, `CommonForm`, `CommonCommand`, `CommonAttribute`,
`FunctionalOption`, `SettingsStorage`, `FilterCriterion`, `ScheduledJob` …),
`Attribute`, `TabularSection`, `Dimension`, `Resource`, `EnumValue`, `PredefinedValue`,
`DefinedType`, `Module`, `Procedure`, `Function`, `Form`, `Command`, `Template`, `Role`,
`EventSubscription`, `Chunk`.

**Рёбра:**
- `CONTAINS` — конфигурация/подсистема → объект.
- `HAS_ATTRIBUTE` / `HAS_TABULAR_SECTION` / `HAS_DIMENSION` / `HAS_RESOURCE` / `HAS_ENUM_VALUE`.
- `REFERENCES` — реквизит/измерение/ресурс → объект (по типу; для составных типов — несколько рёбер).
- `USES_DEFINED_TYPE` — реквизит → `DefinedType`; `DEFINED_AS` — `DefinedType` → объекты.
- `BELONGS_TO_SUBSYSTEM` — объект → подсистема.
- `HAS_MODULE` / `HAS_FORM` / `HAS_COMMAND` / `HAS_TEMPLATE` — объект → дочерние.
- `HAS_RIGHT_ON` — `Role` → объект (права как свойства: Read/Insert/Update/Delete/View/Edit…).
- `WRITES_TO` — документ → регистр (движения).
- `DECLARES` — `Module` → `Procedure`/`Function`.
- `CALLS` — `Procedure`/`Function` → `Procedure`/`Function`; свойства `confidence` (high/medium/low),
  `resolution` (стратегия), опц. `unresolved=true`.
- `HANDLES` — событие формы/команды/объекта → процедура-обработчик.
- `SUBSCRIBES` — `EventSubscription` → типы-источники; `HANDLED_BY` — подписка → процедура.
- `HAS_CHUNK` — узел → `Chunk` (его векторизованное представление).

**Индексы:** уникальные по `(tenant_id, config_id, fqname)`; full-text (Lucene) по
именам/синонимам/коду; vector index по эмбеддингам `Chunk` (cosine, HNSW).

## 7. Парсинг XML (формат Конфигуратора)

Раскладка каталога выгрузки:
- Корень: `Configuration.xml` (свойства конфигурации, дочерние объекты, подсистемы),
  `ConfigDumpInfo.xml` (версии/хеши объектов → инкрементальность).
- Папки по типам: `Catalogs/`, `Documents/`, `Enums/`, `InformationRegisters/`,
  `AccumulationRegisters/`, `CommonModules/`, `Roles/`, `Subsystems/`, `DefinedTypes/`,
  `EventSubscriptions/`, `ExchangePlans/`, `CommonForms/`, `Reports/`, `DataProcessors/`, …
- Объект `<Тип>/<Имя>.xml`: `<Properties>` (Name, Synonym, Comment…), `<ChildObjects>`
  (Attribute, TabularSection, Dimension, Resource, EnumValue, Form, Command, Template…).
- Модули BSL (`*.bsl`): `CommonModules/<N>/Ext/Module.bsl`;
  `<Тип>/<N>/Ext/{ManagerModule,ObjectModule,RecordSetModule,ValueManagerModule}.bsl`;
  формы `<Тип>/<N>/Forms/<F>/Ext/Form/Module.bsl` (+ структура `…/Ext/Form.xml`);
  команды `…/Commands/<C>/Ext/CommandModule.bsl`.
- Права: `Roles/<N>/Ext/Rights.xml` (объект → набор прав).
- Подсистемы: `Subsystems/<N>.xml` → `<Content>` (состав объектов).
- Подписки: `EventSubscriptions/<N>.xml` → `<Source>` (типы) + `<Handler>` (ОбщийМодуль.Метод).

Особенности парсера:
- **Namespace-aware** (lxml): `MDClasses`, `v8` (core), `xs` (XMLSchema), `xcf/readable`,
  `roles`, конфиг-типы. QName → (ns, local).
- **Типы реквизитов**: `<Type>` c одним/несколькими `<v8:Type>`; ссылочные — `…Ref.<Имя>`
  (`CatalogRef`, `DocumentRef`, `EnumRef`, `ChartOfCharacteristicTypesRef`…), примитивы — `xs:string`
  и т.п. с квалификаторами (длина/точность/дата). Составные типы → набор + несколько рёбер `REFERENCES`.
- **Определяемые типы** (`DefinedType`) — раскрытие в реальные типы.
- **Предопределённые** значения (справочники/ПВХ/ПС) — отдельные файлы `Ext/Predefined.*` в новых
  версиях либо внутри объекта; извлекаем имена/коды/значения.
- **Версии формата 8.3.x** — устойчивый разбор, неизвестные элементы не валят импорт (warning).
- **Расширения (.cfe)** — отдельный `config_id=ext:<имя>`; объекты помечаются ролью
  (заимствованный/собственный/переопределённый); связь с базовой конфигурацией ребром.

## 8. BSL-анализ и граф вызовов

- **Парсер:** tree-sitter-bsl через py-tree-sitter → AST; извлечение процедур/функций,
  экспортности, директив компиляции (`&НаКлиенте`/`&НаСервере`), областей, докстрингов.
- **Чанкинг кода:** по границам процедур/функций (см. §9, AST split-then-merge).
- **Разрешение вызовов** (ребро `CALLS` со свойством `confidence`):
  - `ОбщийМодуль.Метод()` → CommonModule по имени + метод → **high**.
  - Локальный `Метод()` в том же модуле (есть в `DECLARES`) → **high**.
  - `Справочники.<Имя>.Метод()` / `Документы.<Имя>…` → метод модуля менеджера → **medium**.
  - Экспортный метод объектного модуля через переменную-объект → **medium/low**.
  - Голый `Метод()`, не найден локально → поиск среди экспортных общих модулей → **low**.
  - Не разрешён → ребро с `unresolved=true` к узлу-заглушке (имя метода сохраняется).
- **Обработчики форм/команд/подписок** связываются ребром `HANDLES`/`HANDLED_BY` по имени метода.

## 9. Стратегия векторизации (бест-практисы)

Основано на разборе аналогов (FSerg, metadata1c/documents1c, metacode) и code-RAG (cAST,
contextual retrieval, hybrid+RRF).

**Единицы индексации (`:Chunk`, поле `kind`):**
- `metadata_object` — карточка объекта: имя + синоним + комментарий + назначение + список реквизитов с типами.
- `attribute` — карточка реквизита/измерения/ресурса: `<путь> ▸ <синоним> (<имя>) : <тип, вкл. составной/ссылочный>`.
- `enum_value` / `predefined` — значения перечислений и предопределённых.
- `code` — процедура/функция (границы AST) + докстринг + контекст-префикс `<Объект>.<Модуль>.<Метод>(экспорт)`.

**Практики:**
1. **Контекст-префикс в каждый чанк** («хлебные крошки»: путь/подсистема/синоним) — фрагмент без контекста бесполезен.
2. **AST-чанкинг кода** (cAST): top-down split-then-merge, бюджет по **не-пробельным символам**, уважение границ процедур; крупные процедуры дробятся, мелкие соседи сливаются.
3. **Мульти-вектор**: `v_ident` (имя+FQN) × `v_semantic` (синоним+комментарий+тип/назначение); для кода `v_signature` × `v_body`. Слияние через RRF.
4. **Гибридный поиск**: vector index ∥ full-text (Lucene) → **RRF** (`1/(k+rank)`), `k≈60`.
5. **Доменная токенизация** для FTS: разбиение `ВерхнийРегистр`/`CamelCase` (`ОбщийМодуль.Метод` → токены), `&НаКлиенте`, `#Если`, GUID, версии платформы.
6. **GraphRAG-достройка**: вектор находит опорные узлы → Cypher достраивает окрестность (реквизиты объекта, вызовы процедуры, права) → возвращаем не голый чанк, а контекст.
7. **Реранкер** (опц., off по умолчанию): cross-encoder `bge-reranker-v2-m3` по топ-50–100; дешёвая альтернатива — 60% вектор + 40% BM25.
8. **Точность > полнота**: мелкие самодостаточные чанки, точный top-k.
9. **Эмбеддить ёмко, детали — из графа**: вектор по короткой карточке, полная структура отдаётся из Neo4j.

**Модель и хранение:** по умолчанию локальная **Qwen3-Embedding** (4B; 0.6B на слабом железе)
или **BGE-m3** (dense+sparse «из коробки»). Облако — OpenAI `text-embedding-3-large` / Voyage за
интерфейсом провайдера. Кеш эмбеддингов запросов по SHA256. Все модели ≤ 4096 dim (лимит Neo4j).
Языки: рус + англ (многоязычная модель + два FTS-анализатора).

## 10. MCP-инструменты (с привязкой к ролям SDLC)

| Инструмент | Назначение | Роли |
|---|---|---|
| `index_configuration(path, config_id, incremental)` | разобрать выгрузку → Neo4j | — (админ/CI) |
| `semantic_search(query, kinds, top_k)` | векторный поиск | БА, разработчик |
| `hybrid_search(query, kinds, top_k)` | вектор + FTS + RRF (+ опц. rerank) | разработчик, СА |
| `get_object(name)` | карточка: реквизиты/типы/формы/модули/права | СА, архитектор |
| `list_metadata(kind, filter)` | дерево/список объектов | БА, тимлид |
| `get_dependencies(object, direction, depth, edge_types)` | обход графа зависимостей | архитектор |
| `impact_analysis(object)` | кто пострадает при изменении | архитектор, ревьюер |
| `find_callers(proc)` / `find_callees(proc)` | граф вызовов | разработчик, ревьюер |
| `call_path(from, to)` | путь вызова между процедурами | архитектор, ревьюер |
| `find_type_usages(typeName)` | где используется тип/объект как реквизит | СА |
| `get_object_rights(object\|role)` | права ролей на объект | СА, тимлид |
| `subsystem_content(name)` | состав подсистемы | БА |
| `cypher_query(query)` | read-only Cypher (whitelist) | архитектор |

Все инструменты автоматически ограничены контекстом арендатора (см. §5).

## 11. Конвейер индексации

1. Прочитать `ConfigDumpInfo.xml` → карта объектов и хешей.
2. **Инкрементально**: сравнить хеши с сохранёнными в Neo4j → переиндексировать только изменённые
   объекты (удалить их подграф + чанки, построить заново).
3. Парсинг метаданных → доменные модели → узлы/рёбра (батчи `UNWIND`).
4. BSL-анализ → процедуры/функции/вызовы → рёбра `DECLARES`/`CALLS`/`HANDLES`.
5. Чанкинг + эмбеддинги → запись векторов на `Chunk` + (пере)создание индексов.
6. Лог сводки: объектов/рёбер/чанков, доля `unresolved` вызовов, время.

## 12. Стек и зависимости (черновик)

- `mcp` (FastMCP) — сервер, Streamable HTTP + stdio.

- `pytest` — тесты. Управление зависимостями: `uv`/`pip`.
- Инфра: `docker-compose` с Neo4j (Community, плагины при необходимости).

## 15. Риски и митигации

- **Разрешение вызовов BSL** (нетривиально) → стратегия с `confidence`, заглушки для неразрешённых,
  итеративное улучшение эвристик; метрика доли `unresolved`.
- **Разнообразие версий/нюансов формата** → устойчивый namespace-aware парсер, неизвестное → warning, тесты на реальной выгрузке.
- **Большие конфиги (ERP/УХ)** → инкрементальность по хешам, батч-запись, lazy-эмбеддинг.
- **Утечка между арендаторами** → tenant-фильтр из контекста запроса (не из аргументов), тесты изоляции.
- **Vector + tenant-фильтр в Neo4j** → filtered vector search / over-fetch+post-filter; при росте — индекс на конфиг.
- **Ресурсы под локальные эмбеддинги** → выбор размера модели (0.6B/4B), облако как fallback.
