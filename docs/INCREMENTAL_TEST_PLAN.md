# План ручного тестирования инкрементальной индексации

Статус: **ожидает выполнения** (нужны реальные правки в Конфигураторе + выгрузка).

## Зачем

Инкрементальные `index` / `vectorize` / `callgraph` определяют «что изменилось» по хешам
`configVersion` из `ConfigDumpInfo.xml`. Нужно на **реальных** изменениях подтвердить, что:
1. каждый вид изменения обнаруживается и обрабатывается точечно;
2. **входящие связи от неизменённых объектов сохраняются**;
3. удаления объектов корректно вычищают граф;
4. — и закрыть **главный открытый вопрос** (см. ниже) про код модулей.

## Метод (для каждого сценария)

1. Внести изменение в Конфигураторе.
2. **Выгрузить конфигурацию в файлы** в тот же каталог выгрузки (Конфигурация → Выгрузить
   конфигурацию в файлы). Это перезапишет `ConfigDumpInfo.xml` и только изменённые `*.xml`.
3. Прогнать соответствующий инкремент:
   `uv run onec-vecgraph index <путь> --tenant-id <t> --incremental`
   (затем `vectorize --incremental`, `callgraph --incremental`).
4. Сверить фактический результат с ожидаемым (колонка «Проверка»).

## ⚠️ Обязательные условия выгрузки

- **`ConfigDumpInfo.xml` должен быть свежим** — именно он несёт новые `configVersion`. Если выгрузить
  только изменённые объекты без обновления `ConfigDumpInfo.xml`, изменения **не обнаружатся**.
- Изменённый объект выгружать **целиком**: `<Тип>/<Имя>.xml` + подпапку `<Имя>/` (модули, формы).
- Для теста **удаления объекта** файл объекта должен **реально исчезнуть** из каталога выгрузки
  (полная выгрузка в тот же каталог это обеспечивает; ручное копирование только изменённых — нет).
- Снимать выгрузку лучше в отдельный каталог-копию на каждый шаг (или фиксировать состояние), чтобы
  можно было сравнивать «до/после».

## 🎯 Главный открытый вопрос — решающий тест **T-CODE-OBJ**

Меняется ли `configVersion` *объекта* (`Catalog.X`) при правке **ТОЛЬКО** кода его
`ObjectModule.bsl` / `ManagerModule.bsl` (без изменения структуры/реквизитов)?
- **Если ДА** — текущий инкрементальный `callgraph` (по `configVersion`) ловит правки кода. ОК.
- **Если НЕТ** — `callgraph --incremental` **пропустит** правку кода в объектном модуле
  → нужно перевести детектор графа вызовов на **хеш содержимого `.bsl`** (callgraph всё равно
  читает файлы, посчитать sha256 ничего не стоит). Для общих модулей правки кода ловятся всегда
  (у них кроме кода ничего нет → `configVersion` меняется).

Как проверить: после T-CODE-OBJ открыть `ConfigDumpInfo.xml`, найти запись
`<Metadata name="Catalog.X" configVersion="...">` и сравнить хеш с предыдущей выгрузкой.

## Матрица сценариев

### A. Структура метаданных (ожидаем: index обновляет, vectorize переэмбеддивает карточку)

| ID | Изменение в Конфигураторе | Ожидание | Проверка |
|----|---------------------------|----------|----------|
| A1 | Изменить **синоним** справочника | index: объект в `changed`; vectorize: его чанки переэмбеддены | `show` → новый синоним; чанк `config_version` обновлён; `search` по новому синониму находит |
| A2 | Изменить **комментарий** объекта | то же (комментарий входит в карточку) | `show` → comment; чанк обновлён |
| A3 | **Добавить реквизит** со ссылочным типом (`СправочникСсылка.Y`) | index: новый `Field` + `REFERENCES→Y`; vectorize: новая карточка реквизита | `show` → реквизит и тип; `find_type_usages Y` содержит объект |
| A4 | **Изменить тип** реквизита (Строка→`ПеречислениеСсылка.Z`; или сделать составным) | `REFERENCES` пересобраны; карточка реквизита (тип) переэмбеддена | `show` → новый тип; `find_type_usages Z` |
| A5 | **Переименовать** реквизит | старый `Field` удалён, новый создан | `show` → нет старого, есть новый |
| A6 | **Удалить** реквизит | `Field` и его `REFERENCES` удалены | `show` → реквизита нет; `find_type_usages` цели уменьшилось |
| A7 | Добавить/изменить **табличную часть** и её реквизиты | `TabularSection` + поля; `tabular_attribute` чанки | `show` → ТЧ и поля; `search` по реквизиту ТЧ находит |
| A8 | Добавить **значение перечисления** / изменить его синоним | `EnumValue` + `HAS_ENUM_VALUE`; `enum_value` чанк | `show` перечисления; `search` по значению |
| A9 | Добавить/изменить **предопределённый** элемент справочника | `Predefined` + `HAS_PREDEFINED`; `predefined` чанк | `show` → предопределённые; (проверить, что меняется `configVersion` объекта) |
| A10 | Изменить **владельца** справочника (Владельцы) | `OWNED_BY` пересобрано | `show` → owners |

### B. Новые и удалённые объекты

| ID | Изменение | Ожидание | Проверка |
|----|-----------|----------|----------|
| B1 | **Создать новый справочник** (2 реквизита, 1 ссылочный) | index: новый объект+подграф; vectorize: новые чанки; callgraph: рутины если есть модуль | `show`/`search` находят новый объект |
| B2 | **Удалить объект целиком** (файл исчезает из выгрузки) | index: `deleted` → объект, дети, чанки, рутины удалены; входящие рёбра от других исчезают вместе с целью | объект не находится; ссылавшиеся объекты целы |
| B3 | **Создать новый общий модуль** с экспортной процедурой | новый объект; после callgraph — рутины | `show`; `find_callees`/`callers` |
| B4 | Новый объект **ссылается на существующий** (реквизит типа `СправочникСсылка.Существующий`) | `REFERENCES` от нового к существующему | `find_type_usages Существующий` содержит новый |

### C. Код BSL (ключевая группа — про configVersion)

| ID | Изменение | Ожидание | Проверка |
|----|-----------|----------|----------|
| C1 (T-CODE-COMMON) | В **общем модуле**: добавить процедуру + вызов другого общего модуля | `configVersion` общего модуля меняется; `callgraph --incremental` ловит (сейчас → fallback на полный, т.к. изменён CommonModule) | `find_callees` новой процедуры показывает вызов; число рутин/CALLS выросло |
| **C2 (T-CODE-OBJ)** | В **`ObjectModule`/`ManagerModule` справочника**: добавить процедуру + вызов общего модуля (структуру НЕ трогать) | **Решающий**: проверить, изменился ли `configVersion` объекта | сравнить хеш `Catalog.X` в `ConfigDumpInfo.xml` до/после; затем `callgraph --incremental` → должна появиться новая рутина/CALLS |
| C3 | В **модуле формы**: добавить процедуру/вызов | у формы свой `configVersion`; проверить, ловит ли инкремент | как C2 |
| C4 | **Переименовать** экспортную процедуру общего модуля, которую вызывают другие | старые `CALLS` к ней исчезают/переразрешаются; fallback→full обязан перестроить корректно | `find_callers` старого имени = 0; нового = прежние callers |
| C5 | **Удалить** процедуру общего модуля | рутина и её `CALLS` исчезают; вызывавшие → unresolved | `find_callers`/`callees`; рутина не находится |

### D. Кросс-объектные связи и сохранение входящих

| ID | Изменение | Ожидание | Проверка |
|----|-----------|----------|----------|
| D1 | Изменить объект **A**, на который ссылается **неизменённый B** | после инкремента `B —REFERENCES→ A` **сохранено** (keep-node) | `find_type_usages A` всё ещё содержит B |
| D2 | **Добавить объект в подсистему** | меняется `configVersion` **подсистемы** (не объекта-члена); index пересобирает подсистему и `CONTAINS` | `get_object(член).in_subsystems` содержит подсистему |
| D3 | **Изменить права роли** (добавить право на объект) | меняется `configVersion` **роли**; `HAS_RIGHT_ON` обновлены | `impact_analysis`/`get_object_rights` показывает роль |
| D4 | Изменить **подписку на событие** (источник/обработчик) | меняется `configVersion` подписки; `SUBSCRIBES`/`HANDLED_BY` обновлены | `show` подписки; `impact_analysis` источника |

### E. Регресс / идемпотентность

| ID | Действие | Ожидание |
|----|----------|----------|
| E1 | Прогнать инкремент **дважды без изменений** | второй раз `changed=0`, `stale=0`, callgraph `stale_objects=0` (для одно-базовой конфигурации) |
| E2 | `--reset` (полный) vs накопленный инкремент | граф/чанки/рутины идентичны полному прогону |

## Конкретные тест-кейсы на ERP/УХ — РАУНД 1 (одна выгрузка)

Каждый кейс — на **отдельном** объекте (чтобы `snapshot-diff` однозначно сопоставил каждый
сменившийся хеш с кейсом). Все правки внести в одном сеансе → одна «Выгрузить в файлы» в
`H:\1C\xml\ERP UH`. Все изменения **аддитивные/безопасные** (без удалений/переименований).

| # | Объект | Что изменить | Что проверяем |
|---|--------|--------------|---------------|
| 1 | `Справочник.Номенклатура` | **только код модуля менеджера**: добавить `Процедура Тест_Инкремент() Экспорт` с вызовом любой процедуры общего модуля; структуру НЕ трогать | **T-CODE-OBJ (менеджерский модуль)**: сменился ли `configVersion` объекта от правки кода |
| 2 | `Справочник.ДоговорыКонтрагентов` | **только код модуля объекта**: добавить процедуру + вызов общего модуля | **T-CODE-OBJ (объектный модуль)** |
| 3 | `ОбщийМодуль.ВзаиморасчетыСервер` | **код**: добавить экспортную процедуру + вызов другого общего модуля | CommonModule → configVersion меняется; callgraph (fallback→full), новые рутины/CALLS |
| 4 | `Справочник.Организации` | **синоним** | vectorize переэмбеддит карточку |
| 5 | `Справочник.СоглашенияСКлиентами` | **добавить реквизит** ссылочного типа (напр. `СправочникСсылка.Валюты`) | новый `Field` + `REFERENCES` + карточка реквизита |
| 6 | `Справочник.ВидыНоменклатуры` | **сменить тип** существующего реквизита (строку→ссылку или сменить цель ссылки) | `REFERENCES` пересобраны; тип в карточке |
| 7 | `Справочник.Лоты` | **добавить табличную часть** с 2 реквизитами (1 ссылочный) | `TabularSection` + `tabular_attribute` чанки |
| 8 | `Перечисление.ОбъектыФинансовогоУчета` | **добавить значение** | `EnumValue` + чанк |
| 9 | `Справочник.ДрайверыОборудования` | **изменить предопределённый** (наименование одного предопределённого) | меняет ли правка предопределённого `configVersion` объекта; `predefined` чанк |
| 10 | `Роль.БазовыеПраваБП` | **добавить право** (напр. Просмотр) на любой объект | configVersion роли; `HAS_RIGHT_ON` |
| 11 | `Подсистема.CRM` | **изменить состав** (добавить существующий объект в подсистему) | меняется configVersion **подсистемы** (не члена); `CONTAINS` пересобран, член не тронут (входящее сохранено) |
| 12 | `ПодпискаНаСобытие.АктуализироватьСтруктуруПредприятия` | **изменить обработчик** (или источник) | configVersion подписки; `HANDLED_BY`/`SUBSCRIBES` |
| 13 | `Справочник.ФизическиеЛица`, любая форма (напр. `ФормаЭлемента`) | **изменить код модуля формы** | у формы свой `configVersion`; ловит ли инкремент правку кода формы |
| 14 | **Контроль:** `Справочник.Валюты` | **НЕ трогать** | должен остаться `unchanged` в диффе и не переэмбеддиться |

После выгрузки: `snapshot` → `snapshot-diff` (увидим, какие из объектов #1–13 сменили хеш и **сменился
ли хеш самого объекта при правке только кода/формы** — это и есть ответ T-CODE-OBJ), затем
`index/vectorize/callgraph --incremental` и сверка.

## Действия по итогам

- Если **T-CODE-OBJ** показывает, что `configVersion` объекта **не меняется** при правке кода его
  модуля → перевести `callgraph --incremental` на **хеш содержимого `.bsl`** (хранить sha256 модуля на
  узле `Module`, сравнивать при инкременте). Это сделает инкремент графа вызовов корректным по коду
  независимо от `configVersion`.
- Если для одно-базовой конфигурации в E1 наблюдается «дрожание» (changed>0 без изменений) — выяснить
  причину (вложенные подсистемы без `configVersion`, объекты без хеша) и при необходимости перейти на
  учёт версий per-(tenant, config) либо на хеш файла объекта.
