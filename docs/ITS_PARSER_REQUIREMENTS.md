# Требования к внешнему парсеру 1С:ИТС (контракт с onec-vecgraph)

> Статус: **СПЕЦИФИКАЦИЯ (к реализации позже)**. Сам парсинг ИТС живёт **вне** этого сервиса
> (может быть на другом стеке/инструменте). onec-vecgraph только потребляет его вывод через
> адаптер `its`. См. отложенный план мульти-источников в `docs/STATE.md` §11.

## 1. Назначение и границы

Инструмент превращает контент 1С:ИТС (методическая поддержка, документация платформы и типовых
решений) в **нормализованный промежуточный формат** для ингеста в onec-vecgraph.

**Делает:** извлечение, очистку, нормализацию, разбиение на логические разделы, проставление
метаданных и (по возможности) подсказок связи с объектами конфигурации.

**НЕ делает (это сервис):** эмбеддинги, запись в Neo4j, семантическую линковку, чанкинг ниже уровня
раздела. Парсер не знает про Neo4j и векторы.

## 2. Вход

- Источник ИТС — на усмотрение инструмента (HTML-выгрузка, локальные файлы, внутренний API).
- **Проектный срез:** выгружать только релевантные проекту разделы/конфигурации (ИТС — проектный,
  per-tenant; общий корпус даёт шум). Состав среза задаётся параметром инструмента.

## 3. Выход (КОНТРАКТ — это и есть граница)

- Каталог файлов, **по одной единице на логический раздел статьи** (не статья-монолит и не каждый
  абзац). Формат — JSON-файл на единицу (или JSONL), кодировка **UTF-8**, корректная кириллица.
- Каталог **коммитится в приватный git-репозиторий** (сервис его клонирует — все источники
  подключаются как git-репозитории). Раскладка: один репозиторий на проект, либо общий с подпапками
  `its/<project>/…`.

### Поля единицы
| Поле | Обяз. | Назначение |
|---|---|---|
| `id` | да | стабильный уникальный id раздела (external_id; для дедупа и инкремента) |
| `title` | да | заголовок раздела |
| `section_path` | да | хлебные крошки (раздел → подраздел → …) — контекст-префикс чанка |
| `text` | да | очищенный текст (без навигации/меню/рекламы; markdown допустим; таблицы → текст/markdown) |
| `version_hash` | да | хеш содержимого раздела; меняется ТОЛЬКО при изменении контента → инкремент |
| `related_fqns` | нет | распознанные точные fqn (`["Document.РеализацияТоваров"]`) → точные рёбра `MENTIONS` |
| `related_kinds` | нет | виды метаданных раздела (`["Document","AccumulationRegister"]`) |
| `source_url` | нет | ссылка на оригинал в ИТС (атрибуция/возврат к источнику) |
| `lang` | нет | `ru` / `en` |
| `product` / `config` | нет | к какой конфигурации/версии платформы относится |
| `tags`, `updated_at` | нет | доп. метаданные |

## 4. Требования к содержимому

- **Чистый текст:** без HTML-обвязки; сохранять смысловую структуру (заголовки → `section_path`).
- **Гранулярность:** одна единица = один логический раздел (точность поиска). Финальный чанкинг —
  на стороне сервиса, но крупные статьи лучше уже разбить по разделам.
- **Стабильные `id` между запусками** — иначе инкремент ломается и появляются дубли.
- **Идемпотентность:** повторный парсинг неизменного раздела → тот же `id` и `version_hash`.

## 5. Доставка и версионирование

- Вывод складывается в git-репозиторий, указанный в манифесте источников проекта
  (`type: its`, `repo`, `branch`, `globs`).
- Версионирование двойное: git-история (аудит) + `version_hash` на единицу (инкрементальный ингест:
  переэмбеддятся только изменённые разделы).

## 6. Лицензия / безопасность

Контент ИТС — **проприетарный** (платная подписка 1С). Репозиторий с выводом — **приватный**, доступ
ограничен, не тиражировать наружу; данные живут в tenant проекта. Не включать в публичную/demo-выгрузку.

## 7. Открытые вопросы (решить при реализации)

- Кто проставляет `related_fqns`/`related_kinds` — парсер (точнее) или сервис семантикой (recall)?
  Желательно — парсер, где может (по явным упоминаниям в тексте ИТС).
- Источник `id` (внутренний id ИТС vs стабильный хеш URL+section_path).
- Нужен ли срез ИТС по составу подсистем проекта на стороне парсера или на стороне ингеста.
